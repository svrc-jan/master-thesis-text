\chapter{Methodology}

\section{Problem description}
Most of the problem description is taken from the \emph{DISPLIB format specification} \cite{kloster2025displibspec}, a\ concise explanation is in accompanied paper \cite{kloster2025displiblibrarytraindispatching}.

Each instance in the \emph{DISPLIB} library consist of set of trains and an objective to be minimized. All data is presented in JSON format \cite{pezoa2016foundations}. Train is defined by an operation graph which is DAG, operation has the following content:

\begin{itemize}
    \item minimal duration
    \item start lower bound (optional)
    \item start upper bound (optional)
    \item list of successors
    \item list of required resources
    \begin{itemize}
        \item resource name
        \item release time
    \end{itemize}
\end{itemize}    

Data \ref{data:op} show and example of an operation entry in the instance JSON.

All operation graphs have one entry (start) operation, with no predecessors, and one exit (end) operation, with no successors. Each operation ends exactly when the successor operation starts, exit operation has no end. 

Resources are anything that is required by the operation, typically physical sections of the railway network. Resources can only be used by one operation at the time and are released after the end of the operation that uses it plus the defined release time.

Objective is defined by a\ list of objectives components, each with the following content 

\begin{itemize}
    \item train
    \item operation
    \item threshold
    \item coefficient (linear objective)
    \item increment (stepwise objective)
\end{itemize}

\begin{remark}
While the format allows both coefficient and increment to be non-zero, in practice none of the instances mix both in one objective component.
\end{remark}

Data \ref{data:obj} show and example of an objective component entry in the instance JSON.

The total objective $J$ is defined as a\ sum of components for active operations only,

\begin{equation}
J = \sum_{o \in O_{a,J}} c_o \max\{0, t_o - \bar{t}_o\} + d_o H(t_o - \bar{t}_o),
\end{equation}

where $O_{a,J}$ is a\ set of active operations that have an objective component, $t_o$ is start of the operation, $\bar{t}_o$ is the threshold for the operation, $c_o$ is the linear objective coefficient, $d_o$ is the stepwise objective increment, and $H(\cdot)$ is Heaviside step function \cite{abramowitz1972handbook}.

A solution to the problem is a\ sequence of events, each event entry corresponds to start of an operation and contains:
\begin{itemize}
    \item train
    \item operation
    \item time
\end{itemize}

Solution is also presented in a\ JSON format, data \ref{data:events} shows truncated solution example. To reiterate, the end event of an operation is the start event of its successor.

\begin{displaycquote}[Feasible solution requirements in \emph{DISPLIB} format specification]{kloster2025displibspec}
    \begin{itemize}
        \item The sequence of events is given in chronological order, i.e., the time of each event is greater than or equal to the time of all preceding events.
        \item The subsequence of operations applying to each train forms a\ path through that train’s operations graph, starting at the entry operation and ending at the exit operation.
        \item The time value of each start event satisfies the lower and upper bounds on the start time of the operation.
        \item The duration from each start event to the corresponding end event must be greater than or equal to the minimum duration of the corresponding operation.
    \item For any pair of operations $o_1$ and $o_2$, where:
        \begin{itemize}
            \item the start event of $o_1$ precedes the start event of $o_2$ in the global sequence, and $o_1$ and $o_2$ belong to different trains, and
            \item $o_1$ and $o_2$ use a\ common resource $r$,
        \end{itemize}
        the following must hold:
        \begin{itemize}
            \item the end event for $o_1$ precedes the start event for $o_2$ in the global sequence, and
            \item the duration from the end event of $o_1$ to the start event of $o_2$ is greater than or equal to the release time of resource $r$ in $o_1$.
        \end{itemize}
    \end{itemize}
\end{displaycquote}

While the events in the solution have a\ time, their order in the list is still significant, as they define in which order the resources are unlocked and locked. This is to prevent a\ resource swap situation described in section \ref{sec:swap}.

\subsection{Resource swap}
\label{sec:swap}

Consider the situation as in figure \ref{fig:swap_railway}, where 2 trains a\ and B are passing a\ section of 2 resources X and Y in opposite direction. If we were to set events A2 and B2 to the same time, we would not have a\ resource collision on either resources (overlap of intervals for operations A1 and B2 on resource X or A2 and B1 on resources Y) but in reality the trains would collide.

In this situation, resource Y is being acquired by a\ while holding X and X is being required by B while holding Y.

We can detect the resource swaps using an event graph by cycle detection, like in figure \ref{fig:swap_event} where A2 and B2 form a\ cycle.

\section{Problem aggregation}

To effectively model the problem we will aggregate the operations and resources from the initial problem into several interconnected structures.

\subsection{Junctions}
First aggregation is what we call junctions, corresponding to a\ single event of one operation starting and one ending (except for starting and ending operations). Because we always want each operation to have single determined event for start and end, the junction graph is a\ directed multigraph, where each edge corresponds to an operation (as opposed to an operation graph where each operation was a\ vertex), figure \ref{fig:junction} shows and example of an operation graph with corresponding junction graph. We use the name junctions as conceptually, the operations converge and diverge in these points as do real tracks in railway junctions.

\begin{figure}[htbp!]
\caption{Operation graph (left) with corresponding junction graph (right)}
\label{fig:junction}
\begin{center}
\includegraphics[width=.8\textwidth]{img/junction.pdf}
\end{center}
\end{figure}

Aside from more natural representation of using vertices as single events and edges as intervals, which translates to the events graphs, we also reduce the number of required vertices and edges, for the example in figure \ref{fig:junction}, it is from 9 vertices and 11 edges to 7 vertices and 9 edges, that is including the often redundant start and end events.

\subsubsection{Disjoint set}
\label{sec:disj_set}
To aggregate the variables into junctions, we use a\ \emph{disjoint set} \cite{GallerFischer1964}. It is often called the \emph{union-find} data structure, it used to partition elements into non-overlapping (disjoint) sets. It has 2 basic operations:
\begin{itemize}
    \item \emph{Find:} Unary operation, returns to what set item belongs to, usually by returning the representative element.
    \item \emph{Union:} Binary operation, if the 2 elements belong to different sets, it joins them together.
\end{itemize}

In short, each element has a\ parent entry, which is at the start initialized to itself and the representative element of a\ set always has the parent as itself. To get the representative we traverse the tree upwards via parent entries until we find the representative. When we use the union operation on 2 elements from different sets, the representative of one becomes a\ representative of the other (if they belong to the same set nothing happens). 

To optimize the data structure we use path compression (flattening of the parent tree by assigned the parent of the element to their parent when traversing the tree) and union by size (always attaching smaller tree to bigger tree) \cite{Tarjan1975}. Both operations with optimizations are in algorithms \ref{alg:disjset_find} and \ref{alg:disjset_union}.


\begin{algorithm}[float, caption={Disjoint set - find}, label={alg:disjset_find}]
input: element $v$
output: representative $s$

begin
  while $u \neq \mathrm{parent}(u)$
    $\mathrm{parent}(u) \gets \mathrm{parent}(\mathrm{parent}(u))$
    $u \gets \mathrm{parent}(u)$
  end
  $s \gets u$
  return $s$
end
\end{algorithm}


\begin{algorithm}[float, caption={Disjoint set - union}, label={alg:disjset_union}]
input: elements $u$ and $v$
output: none

begin
  $u \gets \mathrm{find}(u)$
  $v \gets \mathrm{find}(v)$
  if $u \neq v$
    if $\mathrm{size}(u) < \mathrm{size}(v)$
      $\mathrm{parent}(u) \gets v$
      $\mathrm{size}(v) \gets \mathrm{size}(v) + \mathrm{size}(u)$
    else
      $\mathrm{parent}(v) \gets u$
      $\mathrm{size}(u) \gets \mathrm{size}(u) + \mathrm{size}(v)$
    end
  end
end
\end{algorithm}


\subsubsection{Procedure}
The procedure to create the junctions is done for each train separately. For each operation of the train we use the union operation on each pair of successors (just consecutive pairs are sufficient due to the nature of the union operation), this makes sure that each operation end will correspond to one junction. Each set in the \emph{disjoint set} structure will then correspond to one junction where the elements of the set are the outgoing operations. We also create an additional junction to represent the end of the last (exit) operation with no outgoing operations.

With this set of junctions, we can assign each operation its starting junction. The first (entry)operation starts at the start (source) junction with no incoming operations. We can assign the ending junction for an operation to the starting junction of its successors, thus also getting the set of incoming operations for each junction, the last operation ends at the additional end (sink) junction  with no outgoing operations.

\subsubsection{Invalid transitions}
We could have a possible situation where the set of outgoing operations for a junction $j$ is not equal to the set of successors for an incoming operation $o_{\mathrm{in}}$, $\outedges(j) \neq \mathrm{succ}(o_{\mathrm{in}})$, or vice versa the set of incoming operations for a junction $j$ is not equal to the set of predecessors for an outgoing operation $o_{\mathrm{out}}$, $\inedges(j) \neq \mathrm{pred}(o_{\mathrm{in}})$. This generates an invalid transition $(o_1, o_2)$, such that $o_1 \in \inedges(j)$ and $o_2 \in \outedges(j)$, but $o_2 \notin \mathrm{succ}(o_1)$ or $o_1 \notin \mathrm{pred}(o_2)$. This case is rare however and only affects a few instances, but should be checked when making routing decisions.

An example of such invalid transition is in figure \ref{fig:junct_invalid}, where the transition (2, 3) in the third junction invalid.

\subsection{Levels}
Second is the aggregation to levels. We group parallel junctions to levels,\footnote{In \cite{Dubach2025handling} they call levels \emph{event clusters}, the level term was taken from Venislav Varbanov.} as junctions can be parallel. This is useful to reduce the number of required vertices for each train in the event graph. An example of a\ junction graph and its corresponding level graph is in figure \ref{fig:level}.

\begin{figure}[htbp!]
\caption{Junction graph (top) and its corresponding level graph (bottom)}
\label{fig:level}
\begin{center}
\includegraphics[width=.5\textwidth]{img/levels.pdf}
\end{center}
\footnotesize \textit{Note:} Optional levels are gray.
\end{figure}

Each level $L$ consists of set of junctions $j \in L$ that are parallel and thus mutually exclusive. The incoming and outgoing set of operations for each level is the union of the incoming and outgoing operations of its junctions, respectively,
\begin{equation}
  \inedges(L) = \bigcup_{j \in L} \inedges(j),
\end{equation}

\begin{equation}
  \outedges(L) = \bigcup_{j \in L} \outedges(j).
\end{equation}

The start (source) level contains only the start (source) junction, and the end (sink) level contains only the end (sink) junction.

\subsubsection{Required and optional levels}
Not all levels need to be passed when traversing the graph. Such levels we call optional, the levels that have to be passed when traversing the graph we call required. In \cite{Dubach2025handling} they use the concept of required levels to generate bound constraints. We use required levels to identify border levels, which are used to create routing sections in \ref{sec:sections}. In the example from figure \ref{fig:level} the grayed out levels are optional, because they can be bypassed by an operation.

\subsubsection{Topological generations}
To aggregate the junctions into levels, we use the concept of \emph{topological generations} \cite{Hagberg:2008:NetworkX}. To create these generations we use adjusted Khan's algorithm for topological sorting \cite{Kahn:1962:Topological}, where we group all junctions (vertices) currently without incoming operations (edges) into a\ level as shown in algorithm \ref{alg:topogen}.

\begin{algorithm}[float, caption={Topological generation algorithm}, label={alg:topogen}]
input: junction graph $G(J,\ O)$, junctions $J$ (vertices), operations $O$ (edges)
output: levels $(L_0,\ \dots,\ L_n)$ (subsets of junctions)

begin
  foreach $j \in J$
    $\mathrm{indeg}(j) \gets \indeg(j)$
  end

  $k \gets 0$
  $Z \gets \{J_0\}$

  while $Z \neq \emptyset$
    $L_k \gets Z$
    $Z \gets \{\}$
    foreach $i \in L_k$
      foreach $(i,\ j)$ in $\outedges(i)$
        $\mathrm{indeg}(j) \gets \mathrm{indeg}(j) - 1$
        if $\mathrm{indeg}(j) = 0$
          $Z \gets Z \cup \{j\}$
        end
      end
    end
    $k \gets k + 1$
  end

  return $(L_0,\ \dots,\ L_n)$
end
\end{algorithm}

\subsubsection{Procedure}
We apply algorithm \ref{alg:topogen} to each train separately. We also assign each operation to each level based on the junctions and assign each operation its starting and ending level.

To get required levels, we first label all levels as required and then iterate over all operations and check which level it bypasses (if any), and mark those as optional.

\subsection{Routes}
Now that we have aggregated distinct operation start and end events into junctions and levels, we aggregate the routing decision into routes. Each route consists of set of operations such that if $o_1$ and $o_2$ are in the same route and $o_1$ is a\ predecessor of $o_2$ then, $o_1$ is the \emph{only} predecessor of $o_2$ and $o_2$ is the \emph{only} successor of $o_1$. This follows the train flow network definition in \cite{fuchs2025optimizing}. An example of how operations are grouped into routes is in figure \ref{fig:route}.

\begin{figure}[htbp!]
\caption{Junction graph (top) and its corresponding route graph (bottom)}
\label{fig:route}
\begin{center}
\includegraphics[width=.5\textwidth]{img/routes.pdf}
\end{center}
\footnotesize \textit{Note:} Non-routing junctions are gray.
\end{figure}

This assignment of routes also generates a\ set of routing junctions (junctions where any route ends or begins), each having a\ routing (flow) constraint
\begin{equation}
\sum_{r \in \rho_{\mathrm{in}}(j)} x_r - \sum_{r \in \rho_{\mathrm{out}}(j)} x_r = \begin{cases}
    1 & \text{if $j$ is a\ start (source) junction} \\
    -1 & \text{if $j$ is an end (sink) junction} \\
    0 & \text{otherwise}
\end{cases},
\end{equation}
where $x_r$ is a\ binary variable determining if route $r$ is used, $\sigma_{\mathrm{in}}(j)$ is a\ set of incoming routes and $\sigma_{\mathrm{in}}(j)$ is a\ set of outgoing routes for junction $j$.

\subsubsection{Procedure}
For each train we iterate over operations in topological order and assign them the same route if it is the only successor of its only predecessor, otherwise we create a\ new route. Procedure follows algorithm \ref{alg:route}.

\begin{algorithm}[float=htbp!, caption={Route assignement}, label={alg:route}]
input: operations $O = (o_0,\ \dots\ o_n)$ in topological order
output: route assignment $r: O \rightarrow \mathbb{Z}_0^+$

begin
  $k \gets 0$
  
  foreach $o \in O$
    if $\{p\} = \mathrm{pred}(o) \land \{o\} = border\mathrm{succ}(p)$
      $r(o) \gets r(p)$
    else
      $r(o) \gets k$
      $k \gets k + 1$
    end
  end
  
  return $r$
end
\end{algorithm}

\subsection{Routing sections}
\label{sec:sections}
To separate what routing decisions are connected to each other and which are independent, we now group routes into sections. Each section starts and ends at a\ border level. Border levels are identified using logic in the procedure subsection. Example of a\ junction graph being split into sections is in figure \ref{fig:section}.

\begin{figure}[htbp!]
\caption{Junction graph with identified sections and levels}
\label{fig:section}
\begin{center}
\includegraphics[width=.75\textwidth]{img/section.pdf}
\end{center}
\footnotesize \textit{Note:} L0, L2, L4, and L7 are border levels. Non-routing junctions are gray.
\end{figure}



\subsubsection{Procedure}
\label{sec:proc_section}
For each train we do the following:
\begin{enumerate}
  \item Identify the levels that all the following conditions and mark them as border levels:
  \begin{enumerate}
    \item The level is required.
    \item All junctions $j \in L$ for level $L$ are routing junctions.
    \item For all incoming operation $o_{\mathrm{in}} \in \inedges(L)$ of level $L$, the successor set of $o_{\mathrm{in}}$ is equal to the outgoing operation set of $L$, $\mathrm{succ}(o_{\mathrm{in}}) = \outedges(L)$.
    \item For all outgoing operation $o_{\mathrm{out}} \in \outedges(L)$ of level $L$, the predecessor set of $o_{\mathrm{out}}$ is equal to incoming operation set of $L$, $\mathrm{pred}(o_{\mathrm{out}}) = \inedges(L)$.
  \end{enumerate}
  \item Assign consecutive sections $(S_0,\ \dots,\ S_n)$ based on border levels.
  \item Add all routes within the section (using operation levels) to the specified section.
\end{enumerate}

Condition (a) ensures that each section starts and ends at a required level. Condition (b) ensures that we do not split a route by the border level. Conditions (c) and (d) ensures that any route from the level can be taken independently of which route was taken to the level and, vice versa, any route to the level can be taken independently of which route was taken from the level.

\begin{remark}
A sufficient set of conditions to replace conditions (c) and (d) is:
\begin{itemize}
  \item The level has one junction.
  \item The single junction has no invalid transitions.
\end{itemize}
This is because of the nature of how we create the junctions in the first place. If a level $L$ has a single junction $j$, then $\inedges(L) = \inedges(j)$ and $\outedges(L) = \outedges(j)$, and there are no invalid transitions in this junction, then the following equations hold,
\begin{equation}
  \inedges(L) = \inedges(j) = \mathrm{pred}(o),\ \forall o \in \outedges(L),
\end{equation}
\begin{equation}
  \outedges(L) = \outedges(j) = \mathrm{succ}(o),\ \forall o \in \inedges(L).
\end{equation}
\end{remark}

\subsubsection{Start and end levels}

\begin{remark}
The start level (source) and end level (sink) are always border levels,
\begin{itemize}
  \item Condition (a): They are always required.
  \item Condition (b): They always have on junction, start (source) or end (sink) junction.
  \item Condition (c): There are no incoming operations for the start (source). The end (sink) has one incoming operation, the exit, which has no successor.
  \item Condition (d): There are no outgoing operations for the end (source). The start (source) has one outgoing operation, the entry, which has no predecessor.
\end{itemize}
\end{remark}

\subsection{Resource chunks}
So far we have only done aggregation based on
